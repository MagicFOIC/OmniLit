# OmniLit 桌面与网页共享架构：暂缓目标与后续重建路线

文档状态：可独立执行的后续任务基线  
基线日期：2026-07-13  
适用范围：`desktop-web-shared-architecture-phase-2.md` 中本轮未继续实施的目标  
当前决策：D 已完成；A、B、C、E、F 暂缓

## 1. 使用方式

这份文档用于在未来任意时间重新启动共享架构建设，不依赖当时的聊天上下文。开始新一轮工作时：

1. 先运行第 4 节基线检查，确认当前代码没有退化。
2. 从第 6 节选择一个或多个工作流；没有明确选择时不得默认同时启动全部目标。
3. 先审计仓库和外部环境，再以一个可回滚纵向切片实施。
4. 每个工作流只有在“完成证据”全部具备时才能标记完成；配置文件、脚本或本地模拟不等于真实上线。
5. 继续保留 Qt/QML 稳定页面，直至对应 React 页面完成真实数据、桌面嵌入、浏览器运行、性能和回退验收。

原长期目标文档仍负责解释总体愿景；本文件是基于当前代码状态重新整理的执行入口。

## 2. 当前已完成基线

### 2.1 共享产品与协议

- `packages/shared-schema` 是桌面、Local Agent、Cloud 和 React 的权威协议源，Python/TypeScript 类型由同一 schema 生成。
- `packages/api-client` 提供统一请求、错误、任务和取消语义。
- `packages/platform-bridge` 隔离浏览器与 Qt 原生能力；网页不会伪造桌面文件路径或系统能力。
- React 已覆盖知识图谱、文献库、研究集合、账户/同步，以及本轮 D 的研究工作空间、统计分析、AI 工作区和业务设置。
- Qt WebEngine 使用 Local Agent 同源页面、请求拦截器注入本地 token，并限制导航到 `/app/` 同源资源；不把 token 放入 URL。

### 2.2 D：主要 React 业务页面

D 已按当前纵向切片完成：

- `ResearchWorkspace`、`ResearchStatistics`、`BusinessSettings`、`ResearchBrief` DTO 已进入共享 schema 和生成类型。
- Local Agent 提供 `/v1/workspace`、`/v1/statistics`、`/v1/settings/business`，并以可取消 `research.brief` 任务提供研究简报。
- 研究工作空间复用 LibraryState 中最多四篇比较文献；统计聚合在 Python 共享业务层完成，React 不复制聚合规则。
- AI 默认执行本地确定性证据编排，并明确标注“不是生成式模型输出、不能替代全文核验”。
- 远程模型只有在用户允许研究内容外发、配置无凭据 HTTPS 端点和模型、且 Local Agent 环境存在 `OMNILIT_AI_API_KEY` 时可用。密钥不写入设置文件、不经 API 返回、不进入网页表单。
- 业务设置使用修订号冲突检测、跨进程锁和原子写入，支持主题、密度、减少动效、高对比度、启动页、默认排序和 AI 边界。
- 浏览器无 Local Agent 时使用明确的演示 fixture；它不是 Cloud 持久化。真实 Cloud 目录、工作空间、统计和设置同步仍属于后续目标。
- 桌面侧新增研究工作空间导航和 allowlist `routeUrl`；共享 Web 未启用或加载失败时回到稳定 QML 文献库。

### 2.3 已存在但不等于外部上线的工程基线

- Cloud API 已有账户、租户、同步、ACL、分享、图谱、团队批注、任务、诊断、数据库迁移和健康检查。
- 已有加密备份 envelope、校验/恢复 CLI、保留策略和调度器。
- 已有 Prometheus 指标与规则、生产 Compose/Nginx/镜像配置和 CI 构建门禁。
- 已有源码级 CycloneDX SBOM、第三方通知生成、严格许可证/依赖检查和正式发布阻断报告。
- 已有更新清单签名、防回退、下载限额、替换回滚，以及 Windows/macOS 正式构建的 fail-closed 入口。
- 已有隐私安全的本地崩溃诊断和网页端显式授权 Cloud 上传。

这些能力目前只证明代码与本地/CI 基线，不证明生产域名、外部监控、不可变异地备份、平台签名或正式商业发行已经完成。

## 3. 不能重建或复制的边界

- 不在 React 中直接读取 SQLite、桌面缓存或本地文件路径。
- 不让 QWebChannel 成为普通业务 CRUD 的第二套 API；业务数据继续经过统一 API Client 和 Local Agent/Cloud API。
- 不在 Cloud 上传本地 PDF、全文、批注、研究问题或诊断，除非对应数据控制已显式开启。
- 不在网页、设置 JSON、日志、任务结果、URL、fixture 或 Git 中保存 API Key、Cloud token、签名私钥、备份密钥或平台证书。
- 不把本地 Compose、占位域名、自签名证书、规则文件、unsigned-smoke 包或 fixture 宣称为生产证据。
- 不删除 Qt/QML 回退，除非共享页面已通过真实数据和桌面真机验收。

## 6. 暂缓工作流

### 6.1 A：真实生产托管与 Cloud 业务数据闭环

目标：让普通浏览器通过真实 HTTPS 域名使用受支持的 Cloud 服务，并补齐 D 页面当前只有 Local Agent/fixture 的 Cloud 数据来源。

实施切片：

1. 选定托管平台、区域、域名、TLS edge、容器注册表和数据库/持久卷方案。
2. 建立 dev/staging/production 环境隔离、secret 注入、最小权限和回滚版本。
3. 在 Cloud API 实现文献目录/详情、研究工作空间、统计和业务设置的租户隔离持久化；决定哪些本地元数据同步、哪些内容永不上传。
4. 让统一 API Client 根据运行环境选择 Local Agent 或 Cloud，而不是在无 Local Agent 时长期依赖 fixture。
5. 完成真实浏览器端到端测试：注册/登录、目录同步、集合、比较工作区、统计、AI 边界、冲突、退出和删除。
6. 发布不可变镜像，记录 digest、来源证明、schema 迁移、备份点和回滚步骤。

完成证据：

- 可访问的 HTTPS staging/production URL 与证书验证结果。
- Cloud API 不直接暴露公网；Origin、CSP、反向代理和内部 metrics 边界通过安全检查。
- 跨租户、权限、配额、同步冲突、删除和隐私开关的集成测试。
- 部署 digest、迁移记录、回滚演练和 D 页面真实 Cloud 数据端到端证据。
- 无 fixture、占位域名或开发密钥被作为生产数据路径。

### 6.2 B：桌面诊断审阅与显式上传

目标：用户能在桌面端查看本地脱敏诊断、逐项选择并明确发送；默认永不上传。

实施切片：

1. 在桌面账户层建立 Cloud 会话传输，不复用 Local Agent token。
2. 提供诊断列表、来源/时间/固定错误类型/指纹预览、删除、全选/取消和发送确认 UI。
3. 上传前再次读取 `shareDiagnostics`；未登录、未授权或授权撤销均 fail closed。
4. 仅发送现有 `DiagnosticReportCreateRequest` 白名单字段，不发送 message、stack、URL、路径、正文或任意 context。
5. 处理离线、超时、401/403/429、重复发送和撤销；本地记录的保留/删除行为必须可解释。

完成证据：

- 默认关闭、逐项审阅、明确确认和撤销即时生效的 UI/集成测试。
- 桌面 Cloud token 与 Local Agent token 隔离，敏感字段拒绝测试通过。
- 失败不丢本地报告，成功状态可追踪但不保存服务凭据或异常正文。
- 隐私文档和账户删除/导出行为更新完成。

### 6.3 C：正式供应链、许可证和最终产物合规

目标：从“源码依赖机器违规为 0”提升到“最终网页镜像与桌面安装包可审计、可复现、经法务确认”。

当前明确阻断：

- 根目录缺少项目 LICENSE 或正式商业许可决策。
- Cloud Python 缺少带哈希的完整传递依赖锁。
- 最终容器基础 OS、PyInstaller/native、PySide6/Qt、QtWebEngine/Chromium 未纳入最终产物 SBOM。
- logo、字体、图标和其他资产的来源/许可审查未关闭。
- 部分上游包缺少随产物归档的许可证正文。
- 尚无漏洞例外审批、修复时限和发布阈值政策。

实施切片：

1. 由负责人/法务确定根 LICENSE、商业分发权和第三方 notice 策略。
2. 生成并验证 Python 全传递哈希锁；依赖升级由自动化更新和审查控制。
3. 对最终镜像与实际冻结桌面目录生成 CycloneDX/SPDX、许可证正文包、漏洞报告和构建 provenance。
4. 建立资产清单、来源、作者、许可、修改状态和分发要求。
5. 固化高危/严重漏洞阻断、例外审批、到期日和重新扫描规则。

完成证据：

- `phase2:compliance --release` 对实际发布候选成功。
- 最终产物而非仅源码的 SBOM、notices、漏洞报告、digest 和 provenance 可归档并相互对应。
- 法务/负责人审查记录、例外清单和到期日齐全。

### 6.4 E：Windows/macOS 正式签名、公证与更新发布

目标：使用真实平台发行身份生成可公开分发、可验证、可安全更新的桌面产物。

前置：C 已关闭正式发布阻断；已有更新签名链和 fail-closed 构建入口继续复用。

实施切片：

1. 获取 Windows Authenticode 证书和 Apple Developer ID/公证能力，确认公司主体与续期责任人。
2. 将凭据放入受保护 CI/HSM/签名服务；分支、tag、审批人和日志脱敏策略最小化。
3. Windows 执行签名、时间戳和 `signtool verify`；macOS 执行 hardened runtime、codesign、notarytool、staple、Gatekeeper 验证。
4. 平台签名完成后再计算更新包 SHA 并签更新清单；发布系统拒绝 unsigned-smoke 或合规未通过产物。
5. 在受支持 Windows/macOS 真机执行安装、首次启动、WebEngine、升级、防回退、失败回滚和卸载验收。

完成证据：

- 平台工具对公开候选包的签名/公证验证输出。
- 受保护 CI 审批和凭据轮换记录，不包含可导出的私钥。
- 两个平台的真机安装与从上一正式版本升级证据。
- 正式更新清单、包 digest、平台签名和来源证明一致。

### 6.5 F：外部事故响应、监控通知与恢复演练

目标：把仓库内指标、规则和备份代码接入真实外部系统，形成有人接收、能演练、可复盘的生产闭环。

实施切片：

1. 接入外部 Prometheus/托管监控、Alertmanager 或等价通知渠道；运维凭据与用户会话隔离。
2. 接入集中结构化日志和追踪，保持低基数、无租户/资源 ID/研究内容/凭据。
3. 定义可用性、延迟、错误率、协作容量、任务成功率、备份新鲜度 SLI/SLO 和错误预算。
4. 指定告警负责人、严重度、通知升级、静默、维护窗和 runbook。
5. 将加密备份复制到异地不可变存储，建立密钥托管/轮换和恢复访问审批。
6. 在隔离环境执行恢复演练：验证 envelope、恢复数据库、运行迁移、启动服务、读取账户/同步/图谱/协作数据，并记录 RPO/RTO。
7. 评估外部崩溃聚合服务；继续只上传既有脱敏 schema，不上传异常正文或研究数据。

完成证据：

- 测试告警确实到达指定接收人，并有确认/升级记录。
- 仪表盘、SLO、runbook、值班责任和一次模拟事故复盘。
- 不可变异地对象的保留/删除策略、密钥恢复流程和成功恢复演练报告。
- 实测 RPO/RTO 满足已批准目标；失败演练产生明确修复项。

## 7. 尚未列为 A/B/C/E/F、但恢复架构时必须重新评估的事项

- D 的真实 Cloud 数据闭环包含在 A；在 A 完成前，普通浏览器的研究工作空间/统计/设置仍是 fixture 演示。
- G6 主图 chunk 约 1.45 MB，需要路由/图引擎拆包、加载性能预算和低端设备测量。
- AI 远程端点目前是用户明确配置的 OpenAI-compatible HTTPS 请求；未来如支持多供应商，应在服务层做适配器、出站 allowlist、配额、审计和可取消语义，不把 SDK/供应商逻辑散落在 React。
- 当前桌面共享业务页默认启用，并受 `OMNILIT_SHARED_WEB_GRAPH=0` 显式回退开关控制；发布前仍应做 PySide6/QtWebEngine 真机测试和崩溃率比较。
- Qt/QML 页面退役只在功能、可访问性、离线、性能、原生桥接和回退证据齐备后单独决策。
- 若重新评估 Tauri，应先收集安装体积、内存、启动时间、WebEngine 稳定性、原生能力成本和团队维护数据；不得仅凭偏好重写。

## 8. 迁移、兼容与回滚规则

- Schema 变更必须先改权威 JSON schema，再生成 Python/TypeScript 类型并运行生成物检查。
- DTO 优先做向后兼容的增量扩展；破坏性变更必须版本化，旧桌面与新 Cloud 的兼容窗口要写入部署计划。
- Local Agent 与 Cloud 不共享 token；Cloud 租户数据与桌面本地数据必须有明确来源标识。
- 数据库迁移必须有备份点、前向迁移验证和应用版本回滚策略；不能依赖降级未知 schema。
- React 页面必须同时覆盖 loading、empty、unavailable、error、conflict、cancelled 和 success 状态。
- 新共享页面先在显式 feature flag 下进入桌面；故障时回到 QML 或明确降级页。
- 每个纵向切片独立记录变更文件、数据迁移、验证命令、已知限制和删除/回滚方法。

## 9. 后续单个工作流的完成定义

一个被重新选择的目标只有同时满足以下条件才算完成：

- 实现覆盖共享 schema、服务、API Client、React/Qt 入口和安全边界中所有适用层。
- 单元、契约、集成、端到端和必要真机测试通过，且完整 Python/Web 基线无回归。
- 外部依赖有真实证据；本地 mock、fixture、占位配置和脚本存在不能替代上线证据。
- 隐私、权限、密钥、日志、删除、导出、配额、取消、冲突和回滚行为已验证。
- 文档更新为当时真实状态，并明确仍未完成的目标。
- 完成当前纵向切片后停下，由用户重新选择下一目标；不得自动扩展到其他暂缓工作流。
