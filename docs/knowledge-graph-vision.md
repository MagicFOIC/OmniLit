# OmniLit 文献知识图谱重构长期目标

## 一、项目愿景

将 OmniLit 当前的知识图谱模块，持续重构为一个现代、清晰、美观、可解释、可扩展的文献知识发现平台。

最终产品应融合以下项目的核心优势，可参考其界面设置：

- Connected Papers：沉浸式关系网络和聚焦式图谱探索
- ResearchRabbit：完整的文献发现工作流和多面板交互
- Litmaps：引文关系与时间演化结合
- Open Knowledge Maps：研究领域和主题簇概览
- ORKG：论文研究问题、方法、数据集、指标和结果的语义建模
- VOSviewer：聚类、密度和共现分析
- CiteSpace：研究热点、关键节点、突现趋势和领域演化分析

产品最终应让用户不仅能“看到论文之间有连接”，还能够回答：

1. 这个研究领域由哪些主要主题组成？
2. 哪些论文、作者和机构最重要？
3. 一项研究方法是如何演化的？
4. 某篇论文继承了哪些工作，又影响了哪些工作？
5. 不同论文使用了哪些方法、数据集和评价指标？
6. 当前研究热点是什么，哪些方向正在快速增长？
7. 用户下一步最值得阅读哪些论文？

------

# 二、核心产品目标

构建一个由五类视图组成、彼此联动的文献知识图谱系统。

## 1. 图谱探索视图

以文献、作者、机构、主题、方法、数据集等实体为节点，以引用、合作、隶属、使用、比较、支持等关系为边。

必须支持：

- 搜索并定位节点
- 从一篇种子文献开始探索
- 一度、二度关系展开
- 引用文献和被引文献切换
- 节点悬停预览
- 节点选中与邻居高亮
- 关系类型筛选
- 年份、主题、作者、机构、期刊筛选
- 社区聚类
- 最短路径
- 保存当前视图
- 分享和导出
- 撤销、重做和恢复默认视图
- 右侧详情面板
- 图谱与文视图
- 右侧详情面板
- 图谱与文献列表双向联动

图谱默认不得一次显示整个数据库，而应采用“焦点加上下文”的渐进式探索模式。

## 2. 研究领域概览视图

以主题气泡、主题簇或层级地图展示一个领域的整体结构。

必须呈现：

- 一级研究主题
- 二级子主题
- 每个主题的文献数量
- 主题代表论文
- 主题代表作者
- 主题增长速度
- 主题之间的相似性
- 用户当前所在主题的位置

该视图用于回答“这个领域整体在研究什么”，不得退化为另一张普通节点关系图。

## 3. 时间演化视图

将引文关系、研究主题和论文发表时间结合起来。

必须支持：

- 按年份排列论文
- 显示关键引文路径
- 显示研究主题的出现、增长、分裂、合并和衰退
- 查看某一主题在不同年份的代表论文
- 播放领域演化过程
- 调整时间范围
- 比较两个主题的发展速度
- 标记里程碑论文和转折点

时间视图需要与图谱视图联动。用户选择一个年份、主题或论文后，其他视图同步更新。

## 4. 聚类与密度分析视图

用于呈现关键词共现、作者合作、文献共被引、文献耦合和研究主题密度。

必须支持：

- 社区聚类
- 聚类标签
- 密度热力图
- 聚类规模比较
- 聚类内部关键论文
- 聚类之间的桥接节点
- 不同分析类型切换
- 聚类结果解释

聚类不能只依赖颜色。还应通过图例、标签、边界、形状或详情说明保证可读性。

## 5. 论文语义比较视图

基于 ORKG 风格的数据模型，将论文内容结构化。

至少支持以下实体：

- Paper
- Author
- Institution
- Venue
- Topic
- ResearchQuestion
- Method
- Model
- Dataset
- Metric
- Result
- Conclusion
- Limitation

至少支持以下关系：

- AUTHOR_OF
- AFFILIATED_WITH
- PUBLISHED_IN
- CITES
- HAS_TOPIC
- ADDRESSES
- USES_METHOD
- USES_MODEL
- USES_DATASET
- EVALUATED_BY
- REPORTS_RESULT
- SUPPORTS
- CONTRADICTS
- EXTENDS
- IMPROVES_ON

用户应能够选择多篇论文，以表格或卡片形式比较：

- 研究问题
- 方法
- 模型
- 数据集
- 指标
- 实验结果
- 贡献
- 局限性

# 三、统一交互设计原则

## 1. 渐进式信息展示

默认只展示当前任务需要的信息。

不同缩放级别展示不同内容：

- 远距离：主题簇、核心论文和整体结构
- 中距离：论文标题、年份、主要作者和关键关系
- 近距离：完整标题、期刊、引用数、方法、数据集和关系标签

不得让全部节点、标签和边同时出现。

## 2. 焦点与上下文

用户选择节点后：

- 当前节点成为视觉焦点
- 相邻节点保持正常亮度
- 无关节点降低透明度
- 相关路径高亮
- 右侧打开详情面板
- 底部或侧边列表同步定位

## 3. 图谱与列表并存

不得要求用户仅通过图形完成所有操作。

图谱旁边必须有可搜索、排序、筛选的文献列表。列表和图谱的选中、悬停、筛选状态应保持同步。

## 4. 动画服务于理解

动画只能用于：

- 节点展开
- 布局切换
- 路径追踪
- 时间演化
- 聚类切换
- 状态变化反馈

禁止无意义的持续漂浮、频繁闪烁和过度发光。

## 5. 视觉一致性

建立统一设计令牌，包括：

- 背景色
- 卡片色
- 节点类型颜色
- 文字层级
- 边线透明度
- 阴影
- 圆角
- 间距
- 动画时长
- 交互状态

同一种实体在所有页面必须使用一致的颜色、图标和命名。

------

# 四、推荐视觉方向

整体采用“专业学术工作台＋沉浸式图谱画布”的设计语言。

推荐结构：

- 顶部：全局搜索、当前研究集合、视图切换和用户操作
- 左侧：筛选器、图层控制和分析类型
- 中间：主图谱或主要可视化
- 右侧：节点、论文或聚类详情
- 底部：时间轴、统计摘要或可折叠文献列表

视觉要求：

- 页面具有清晰的信息层级
- 深色图谱画布可搭配浅色产品框架
- 控制节点颜色数量，避免彩虹式配色
- 默认边线弱化，选择后再增强
- 节点大小差异应经过限制，避免少数节点占满画面
- 标签使用碰撞检测和优先级控制
- 所有视图必须包含清楚的图例和状态说明
- 同时支持浅色和深色模式时，应使用设计令牌实现，不得分别硬编码

------

# 五、数据与算法能力

系统应逐步支持以下分析能力：

- 引用网络
- 被引网络
- 共被引分析
- 文献耦合
- 关键词共现
- 作者合作网络
- 机构合作网络
- PageRank 或类似重要性评分
- 社区发现
- 桥接节点识别
- 最短路径
- 主题相似度
- 文献语义相似度
- 关键词突现
- 主题增长率
- 主题演化路径
- 核心论文识别
- 推荐阅读路径

所有算法结果必须提供可理解的解释。例如，不只显示“重要性 0.86”，还应说明该论文因高被引、连接多个聚类或位于关键路径而被判定为重要。

------

# 六、工程实施原则

Codex 在每一轮工作中必须先检查现有代码、组件、数据结构和接口，不得在未理解项目结构的情况下整体推翻重写。

每轮迭代应遵循以下流程：

1. 检查当前实现和已有功能。
2. 找出阻碍目标实现的最小问题集合。
3. 选择一个可以独立交付的纵向功能切片。
4. 先定义数据和状态，再实现视觉界面。
5. 完成真实交互，不只制作静态演示。
6. 添加或更新测试。
7. 检查性能、空状态、错误状态和加载状态。
8. 确认没有破坏现有功能。
9. 更新开发文档和后续任务。
10. 让项目保持可运行、可测试、可继续开发。

禁止：

- 一次性重写整个前端
- 为追求视觉效果破坏已有业务逻辑
- 创建无法连接真实数据的静态页面
- 在多个组件中重复实现相同状态
- 将节点颜色、关系名称和布局参数分散硬编码
- 仅完成组件文件而不接入产品入口
- 在没有性能策略的情况下渲染全部节点
- 用大量模拟数据掩盖接口问题
- 留下无法编译或无法运行的中间状态

------

# 七、建议阶段里程碑

## 阶段一：建立统一图谱基础

完成：

- 统一节点和关系数据结构
- 图谱数据适配层
- 设计令牌
- 节点类型配置
- 关系类型配置
- 图谱状态管理
- 搜索、筛选和选中状态
- 右侧详情面板
- 基础错误、空白和加载状态

验收结果：

用户可以搜索一篇论文，将其作为中心节点，并查看作者、主题、引用和被引关系。

## 阶段二：完成现代化图谱探索器

完成：

- 聚焦式布局
- 邻居展开
- 语义缩放
- 节点和边高亮
- 图谱与列表联动
- 布局切换
- 小地图
- 图例
- 保存和恢复视图
- 图片导出

验收结果：

形成具有 Connected Papers 视觉感受和 ResearchRabbit 工作流特征的可用图谱探索页面。

## 阶段三：加入领域概览

完成：

- 主题聚类
- 主题气泡地图
- 子主题层级
- 代表论文
- 主题规模和增长指标
- 从主题概览进入局部图谱

验收结果：

用户能够在不阅读大量论文的情况下理解一个研究领域的主要结构。

## 阶段四：加入时间演化

完成：

- 文献时间轴
- 关键引文路径
- 主题随时间变化
- 时间范围筛选
- 演化播放
- 图谱和时间轴联动

验收结果：

用户能够追踪一项技术、方法或主题从早期论文到近期研究的发展过程。

## 阶段五：加入聚类和趋势分析

完成：

- 共被引分析
- 文献耦合
- 关键词共现
- 密度视图
- 关键节点
- 桥接节点
- 突现关键词
- 主题增长趋势
- 结果解释

验收结果：

系统能够生成类似 VOSviewer 和 CiteSpace 的分析结果，但采用更现代、易理解的交互方式。

## 阶段六：加入论文语义知识图谱

完成：

- 研究问题、方法、模型、数据集、指标和结果的数据结构
- 论文语义抽取结果展示
- 语义关系图
- 多论文比较
- 数据来源和置信度
- 人工校正入口

验收结果：

系统从“文献关系网络”升级为能够比较论文研究贡献的知识图谱。

------

# 八、性能目标

设计和实现时必须考虑数据规模，而不是等性能出现问题后再处理。

最低目标：

- 小型子图在常见设备上流畅交互
- 图谱初始只加载必要节点
- 扩展节点时按需请求数据
- 节点和边支持分层加载
- 大量节点时使用聚合、简化或层级细节策略
- 布局计算不得长期阻塞主线程
- 搜索和筛选操作应及时反馈
- 页面切换后保留合理的用户上下文
- 避免因详情面板更新而重新渲染整个图谱

应建立固定测试数据集，对以下规模进行基准测试：

- 100 个节点
- 1,000 个节点
- 5,000 个节点
- 10,000 个节点

超过合理显示范围时，不强行展示全部节点，而应使用聚类、抽样、分页、聚合节点或按需展开。

------

# 九、质量验收标准

每一个功能只有同时满足以下条件才视为完成：

- 使用真实数据或正式接口
- 有加载状态
- 有空状态
- 有错误状态
- 有交互反馈
- 有键盘或基础无障碍支持
- 不破坏已有页面
- 通过现有测试
- 新增关键逻辑测试
- 代码不存在明显重复
- 配置集中管理
- 在不同数据规模下表现合理
- 页面视觉与现有产品保持统一
- 用户能够理解图中颜色、大小、位置和关系的含义
- 文档说明数据来源、算法含义和已知限制

------

# 十、Codex 持续执行规则

每次接到该长期任务后，不要试图一次完成全部目标。

每轮只选择一个最有价值、依赖关系明确、可以独立验收的任务，并交付一个可运行的增量。

每轮开始时：

- 阅读相关代码
- 确认当前完成阶段
- 找到最小可交付改进
- 明确本轮验收条件

每轮结束时输出：

- 本轮解决的问题
- 修改的主要文件
- 新增或调整的数据结构
- 完成的交互
- 测试结果
- 性能影响
- 尚未完成的问题
- 下一轮最优先任务

如果发现架构问题，应优先以兼容方式建立抽象层，再逐步迁移，不得直接进行高风险的大规模重写。

如果需求与现有代码冲突，应优先保证：

1. 项目可以正常运行
2. 用户已有能力不丢失
3. 数据结构逐渐统一
4. 新旧模块可以渐进迁移
5. 每一轮都有可见、可验证的产品进步

------

# 十一、最终完成定义

当用户可以从一个研究主题或一篇种子论文出发，完成以下连续工作流时，该长期目标才算基本完成：

1. 搜索并选择研究主题或种子论文。
2. 查看领域主题地图。
3. 进入论文关系图谱。
4. 展开引用、作者、机构、方法和数据集关系。
5. 通过时间轴查看研究演化。
6. 识别核心论文、聚类、桥接节点和研究热点。
7. 比较多篇论文的方法、数据集、指标和结果。
8. 保存研究集合和当前图谱视图。
9. 获得下一步阅读建议。
10. 理解每项分析结果的来源和含义。

最终产品应同时具备：

- Connected Papers 式的视觉吸引力
- ResearchRabbit 式的研究工作流
- Litmaps 式的时间演化能力
- Open Knowledge Maps 式的领域概览
- ORKG 式的语义知识结构
- VOSviewer 式的聚类与密度分析
- CiteSpace 式的趋势和关键路径识别

但 OmniLit 应形成自己的统一设计语言、数据模型和用户体验，而不是七个项目功能的简单拼接。
