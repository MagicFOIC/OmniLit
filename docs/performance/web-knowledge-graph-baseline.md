# Web 知识图谱性能基线

日期：2026-07-13。环境：Windows、Node.js 24.16.0、Vite 7.3.6、G6 5.1.1。

## 当前切片

本基线只覆盖共享状态筛选、固定小图真实浏览器交互和生产构建体积。尚未用 1k 以上节点
执行 G6 可视渲染，因此不能据此声称大型 Web 图谱帧率或内存已达最终要求。

### 状态筛选

固定测试覆盖 100、1,000、5,000、10,000 节点及最多 `n-1` 条边。每个规模执行关键词、
节点类型和端点完整性筛选，单次门禁为 500 ms；四个规模与其他状态测试所在文件总耗时
约 10 ms。算法保持 O(nodes + edges)，不执行布局或图计算。

### 浏览器交互

固定 GraphData：完整图为 6 个节点、5 条关系、6 种节点类型、2 条证据；首屏只投影中心论文。
真实浏览器已验证：

- G6 Canvas 成功渲染；
- Canvas 节点点击同步详情与可搜索列表选择；
- 类型筛选从 5/4 收敛为 1/0；
- 无匹配搜索显示空状态并可恢复；
- React Strict Mode 下无 G6 销毁竞态、控制台错误或警告。
- 首屏为 1 节点/0 关系；中心论文展开后为 5 节点/4 关系，邻居按 ID 合并且没有重复；
- 文献投影从 1 条更新为 2 条，点击引用文献同步详情、列表选中态和画布聚焦。

开发模式提供 `?benchmark=1000` 的确定性星型图基线入口；自动测试验证它生成 1,000 节点、
999 关系且 ID 唯一。首次真实运行已证明 G6 Canvas 能完成 1,000/999 渲染，同时发现无障碍
列表创建 1,000 个按钮、DOM 快照超过 3,000 行。现已改为每批 100 个的可搜索窗口，并对
250 节点以上图谱只显示高优先级标签；渲染器会报告实际 G6 render 时长和元素数。优化后
优化后真实浏览器复测结果：

- 1,000 节点/999 关系首次 G6 render：391.8 ms；清除筛选后的完整重绘为 503.2 ms；
- 作者类型筛选投影 267 节点：123.1 ms；精确搜索到 1 节点：10.6 ms；
- 无障碍列表同时存在的按钮从约 1,001 个降为 101 个（100 条结果 + “再显示”）；
- Canvas、筛选、搜索和列表仍可交互，控制台无 warning/error，Vite 错误覆盖层不存在；
- 外部浏览器隔离环境没有开放 RAF 和 JS heap 指标。基准页面现内置 1 秒 RAF/可用 heap
  采样，下轮浏览器读取页面输出后才能给出 FPS/内存数字；当前不虚报这两项达标。

Local Agent LOD 继续使用既有固定 100/1k/5k/10k 基准：overview/normal/detail 预算分别为
240/480/900，绝对不超过 1,200 节点；服务契约测试证明 1k 概览产生聚合节点、保持 pin/元数据
且投影未超过 120 ms 内部门禁。5k/10k 仍由 `test_knowledge_graph_lod` 验证确定性、边预算和
线性时间上限。

### 构建体积

- 应用入口 JS：约 244.86 kB，gzip 79.52 kB。
- G6 图谱异步 chunk：约 1,431.95 kB，gzip 415.59 kB。
- CSS：约 12.82 kB，gzip 3.19 kB。

图谱 chunk 不进入普通路由首包，但体积仍是已知限制。下一轮大型图基线应同时记录加载时间、
交互时间、内存和帧率，再决定是否需要更细的 G6 扩展裁剪、Web Worker 布局或后端投影。
